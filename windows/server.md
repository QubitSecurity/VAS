#### `W-01` `상` Administrator 계정 이름 변경 또는 보안성 강화

## [W-01] Administrator 계정 이름 변경 (계정_스크립트)

### 개요
| 항목 | 내용 |
|------|------|
| 점검 코드 | W-01 |
| 항목명 | Administrator 계정 이름 변경 또는 보안성 강화 |
| 위험도 | 상 (High) |
| 범주 | 계정 관리 |

### 점검 목적
Windows 기본 관리자 계정인 `Administrator`는 공격자에게 이미 알려진 계정명입니다. 계정명이 기본값 그대로라면 공격자는 아이디를 알고 있는 상태에서 패스워드만 대입하면 되므로 무차별 대입 공격(Brute Force) 성공 가능성이 크게 높아집니다. 계정 이름을 변경하여 공격자가 유효한 계정명을 추측하기 어렵게 해야 합니다.

### 점검 기준
| 구분 | 기준 | 설명 |
|------|------|------|
| 안전 | Administrator 계정 이름이 변경되어 있는 경우 | 기본 계정명을 사용하지 않아 계정명 유추를 통한 공격을 방어할 수 있다 |
| 취약 | Administrator 계정 이름이 기본값(Administrator)으로 유지되는 경우 | 공격자가 계정명을 이미 알고 있으므로 패스워드 공격만으로 관리자 권한 획득이 가능하다 |


### 참고 문서
- 윈도우 서버는 Administrator 계정을 비활성화할 수 있으나 안전 모드로 컴퓨터를 시작할 경우 본계정은 자동으로 활성화됨 [W-01]

#### `W-02` `상` Guest 계정 비활성화

## [W-02] Guest 계정 비활성화 (계정_스크립트)

### 개요
| 항목 | 내용 |
|------|------|
| 점검 코드 | W-02 |
| 항목명 | Guest 계정 비활성화 |
| 위험도 | 상 (High) |
| 범주 | 계정 관리 |

### 점검 목적
Guest 계정은 인증 없이 또는 최소한의 인증만으로 시스템에 접근할 수 있는 기본 계정입니다. 활성화된 상태로 방치되면 인증 우회를 통한 시스템 접근, 내부 정보 열람 등의 경로가 될 수 있습니다. 불필요한 계정은 반드시 비활성화하여 공격 표면을 줄여야 합니다.

### 점검 기준
| 구분 | 기준 | 설명 |
|------|------|------|
| 안전 | Guest 계정이 비활성화(사용 안 함)되어 있는 경우 | 불필요한 접근 경로가 차단되어 계정 기반 공격 위험이 감소한다 |
| 취약 | Guest 계정이 활성화되어 있는 경우 | 인증 없이 또는 낮은 권한으로 시스템 접근이 가능하여 정보 유출 및 권한 상승 공격에 악용될 수 있다 |


### 참고 문서
- Windows Guest 계정은 삭제 불가능한 built-in 계정으로 보안 강화 목적으로 반드시 비활성화 처리해야 함 [W-02]

#### `W-03` `상` 불필요한 계정 제거

## [W-03] 불필요한 계정 제거 (계정_스크립트)

### 개요
| 항목 | 내용 |
|------|------|
| 점검 코드 | W-03 |
| 항목명 | 불필요한 계정 제거 |
| 위험도 | 상 (High) |
| 범주 | 계정 관리 |

### 점검 목적
퇴직자, 장기 미사용, 테스트 목적으로 생성된 계정 등 업무상 불필요한 계정이 시스템에 잔존할 경우, 공격자가 해당 계정을 탈취하여 시스템에 접근할 수 있습니다. 계정 수명 주기를 관리하여 불필요한 계정을 삭제하거나 비활성화해야 합니다.

### 점검 기준
| 구분 | 기준 | 설명 |
|------|------|------|
| 안전 | 모든 계정이 현재 업무상 필요하며 정기적으로 검토된 경우 | 불필요한 계정이 없어 불법 접근 경로가 존재하지 않는다 |
| 취약 | 퇴직자, 장기 미사용, 용도 불명 계정 등이 활성화 상태로 존재하는 경우 | 해당 계정을 통한 불법 접근 및 권한 탈취 위험이 있다 |



### 참고 문서
- 무차별 대입 공격(Brute Force Attack): 특정 암호를 해독하기 위해 가능한 모든 값을 대입하는 공격 방법 [W-03]

#### `W-04` `상` 계정 잠금 임계값 설정

## [W-04] 계정 잠금 임계값 설정 (계정_스크립트)

### 개요
| 항목 | 내용 |
|------|------|
| 점검 코드 | W-04 |
| 항목명 | 계정 잠금 임계값 설정 |
| 위험도 | 상 (High) |
| 범주 | 계정 관리 |

### 점검 목적
계정 잠금 임계값이 설정되어 있지 않으면 공격자는 로그인 화면에서 수천~수만 번의 패스워드를 제한 없이 시도할 수 있습니다(무차별 대입 공격, Brute Force). 일정 횟수 이상 로그인 실패 시 계정을 잠그도록 설정하면 이러한 공격을 효과적으로 차단할 수 있습니다.

### 점검 기준
| 구분 | 기준 | 설명 |
|------|------|------|
| 안전 | 계정 잠금 임계값이 5회 이하로 설정된 경우 | 일정 횟수 실패 시 계정이 잠겨 무차별 대입 공격을 효과적으로 차단한다 |
| 취약 | 계정 잠금 임계값이 6회 이상이거나 0(잠금 없음)으로 설정된 경우 | 패스워드를 무제한으로 시도할 수 있어 계정 탈취 위험이 크게 높아진다 |


### 참고 문서
- ※계정 잠금 임계값 설정은 사용자 계정이 잠기는 로그온 실패 횟수를 결정하며 잠긴 계정은 관리자가 재설정하거나 해당 계정의 잠금 유지 시간이 만료되어야 사용할 수 있음
※계정 잠금 정책: 해당 계정이 시스템으로부터 잠기는 환경과 시간을 결정하는 정책으로 ‘계정 잠금 기간’, ‘계정 잠금 임계값’, ‘다음 시간 후 계정 잠금 수를 원래대로 설정’의 세 가지 하위 정책을 가짐
※관련 점검 항목: W-08(중) [W-04]

#### `W-05` `상` 해독 가능한 암호화를 사용하여 암호 저장 해제

## [W-05] 해독 가능한 암호화로 암호 저장 해제 (계정_스크립트)

### 개요
| 항목 | 내용 |
|------|------|
| 점검 코드 | W-05 |
| 항목명 | 해독 가능한 암호화를 사용하여 암호 저장 해제 |
| 위험도 | 상 (High) |
| 범주 | 계정 관리 |

### 점검 목적
"해독 가능한 암호화로 암호 저장" 정책이 활성화되면 패스워드가 복호화 가능한 형태로 저장됩니다. 이는 사실상 평문 저장과 동일하며, 공격자가 SAM 데이터베이스나 Active Directory에 접근할 경우 모든 계정의 패스워드를 즉시 획득할 수 있게 됩니다.

### 점검 기준
| 구분 | 기준 | 설명 |
|------|------|------|
| 안전 | 해당 정책이 **사용 안 함**으로 설정된 경우 | 패스워드가 단방향 해시로만 저장되어 복호화를 통한 패스워드 탈취가 불가능하다 |
| 취약 | 해당 정책이 **사용**으로 설정된 경우 | 패스워드가 복호화 가능한 형태로 저장되어 데이터베이스 접근 시 즉시 패스워드 노출 위험이 있다 |


### 참고 문서
- “해독 가능한 암호화를 사용하여 암호 저장” 정책은 암호를 암호화하지 않은 상태로 저장하여 일반 텍스트 버전의 암호를 저장하는 것과 같으나 시스템에서 기본적으로 동작하지는 않음 [W-05]

#### `W-06` `상` 관리자 그룹에 최소한의 사용자 포함

## [W-06] 관리자 그룹에 최소한의 사용자 포함 (계정_스크립트)

### 개요
| 항목 | 내용 |
|------|------|
| 점검 코드 | W-06 |
| 항목명 | 관리자 그룹에 최소한의 사용자 포함 |
| 위험도 | 상 (High) |
| 범주 | 계정 관리 |

### 점검 목적
Administrators 그룹에 불필요하게 많은 계정이 포함될 경우, 그 중 하나만 탈취되어도 시스템 전체에 대한 완전한 제어권을 잃게 됩니다. 최소 권한 원칙(Principle of Least Privilege)에 따라 관리자 그룹에는 반드시 필요한 계정만 포함해야 합니다.

### 점검 기준
| 구분 | 기준 | 설명 |
|------|------|------|
| 안전 | Administrators 그룹에 업무상 필요한 최소한의 계정만 포함된 경우 | 최소 권한 원칙이 적용되어 계정 탈취 시 피해 범위가 제한된다 |
| 취약 | Administrators 그룹에 불필요한 계정이 다수 포함된 경우 | 불필요한 관리자 계정이 공격 경로로 악용될 수 있으며 피해 범위가 확대된다 |



### 참고 문서
- ※관리 권한의 오남용으로 인한 시스템 피해를 줄이기 위해서 관리 업무를 위한 계정과 일반 업무를 위한 계정을 분리하여 사용하는 것이 바람직함
※시스템 관리를 위해서 관리 권한 계정과 일반 권한 계정을 분리하여 운영하는 것을 권고
※시스템 관리자는 원칙적으로 1명 이하로 유지하고, 부득이하게 2명 이상의 관리 권한자를 유지해야 할 경우, 관리자 그룹에는 최소한의 사용자만 포함하도록 해야 함 [W-06]

#### `W-07` `상` 공유 권한 및 사용자 그룹 설정

## [W-07] 공유 권한 및 사용자 그룹 설정 (네트워크_스크립트)

### 개요
| 항목 | 내용 |
|------|------|
| 점검 코드 | W-07 |
| 항목명 | 공유 권한 및 사용자 그룹 설정 |
| 위험도 | 상 (High) |
| 범주 | 공유 및 권한 |

### 점검 목적
네트워크 공유 폴더에 Everyone 그룹이 포함되거나 쓰기 권한이 과도하게 부여된 경우, 인증된 내부 사용자뿐 아니라 인증 우회에 성공한 공격자도 파일을 읽거나 악성코드를 업로드할 수 있습니다. 공유 폴더는 반드시 필요한 사용자와 그룹에만 최소 권한으로 설정해야 합니다.

### 점검 기준
| 구분 | 기준 | 설명 |
|------|------|------|
| 안전 | 공유 폴더에 Everyone 그룹이 없고, 필요한 사용자/그룹에만 최소 권한이 부여된 경우 | 불필요한 접근이 차단되어 정보 유출 및 악성코드 업로드 위험이 감소한다 |
| 취약 | 공유 폴더에 Everyone 그룹이 포함되거나 쓰기 권한이 불필요하게 부여된 경우 | 모든 사용자가 공유 자원에 접근 가능하여 정보 유출 및 시스템 침해 위험이 높아진다 |



### 참고 문서
- "응용프로그램이나 백업 용도로 반드시 공유가 필요한 경우에는 Everyone 권한 대신 특정 관리자 계정이나 서비스 전용 계정을 생성하여 최소 권한으로 할당할 것을 권고함." [W-16]

### 개요
| 항목 | 내용 |
|------|------|

| 위험도 | 상 (High) |
| 범주 | IIS 관리 |

### 점검 목적
IIS(Internet Information Services)는 웹 서비스를 제공하는 서버 소프트웨어로, 불필요하게 구동 중인 경우 외부 공격의 주요 진입점이 됩니다. 웹 서비스가 필요하지 않은 서버에서 IIS가 실행 중이라면 즉시 중지해야 하며, 필요한 경우라도 보안 설정을 강화해야 합니다.

### 점검 기준
| 구분 | 기준 | 설명 |
|------|------|------|
| 안전 | IIS 서비스가 비활성화되어 있거나, 업무상 필요에 의해 운영되며 보안 설정이 적용된 경우 | 불필요한 웹 서비스 노출이 없어 외부 공격 표면이 감소한다 |
| 취약 | 업무상 불필요한 IIS 서비스가 실행 중인 경우 | 불필요한 웹 서버가 외부 공격의 진입점으로 악용될 수 있다 |


### 참고 문서
- 특별한 목적으로 웹 서비스를 사용하는 경우가 아니라면 서비스를 중지하고 '시작 유형'을 '사용 안 함'으로 설정하여 잠재적인 취약점 노출을 차단해야 한다.
- 제어판의 서비스 관리 도구(services.msc)를 통해 해당 서비스의 설치 및 구동 여부를 확인할 수 있다. [W-19]

### 개요
| 항목 | 내용 |
|------|------|
| 점검 코드 | W-65 |

| 위험도 | 중 (Medium) |
| 범주 | 접근 제어 |

### 점검 목적
Telnet은 모든 데이터를 평문으로 전송하는 구형 원격 접속 프로토콜입니다. Telnet을 사용하는 경우 네트워크 스니핑을 통해 자격증명과 세션 데이터 전체가 노출됩니다. Telnet 서비스는 비활성화하고 SSH 또는 RDP(TLS 적용)로 대체해야 합니다.

### 점검 기준
| 구분 | 기준 | 설명 |
|------|------|------|
| 안전 | Telnet 서비스가 비활성화된 경우 | 평문 전송을 통한 자격증명 및 세션 데이터 노출 위험이 차단된다 |
| 취약 | Telnet 서비스가 실행 중인 경우 | 자격증명 및 세션 데이터 전체가 네트워크 스니핑에 노출될 수 있다 |

### 참고 문서
- Windows 서버의 Telnet 서비스의 두 가지 인증 방법
Ÿ NTLM 인증: 암호를 전송하지 않고 negotiate/challenge/response 절차로 인증 수행
Ÿ Password 인증: 관리자 및 Telnet Clients 그룹에 포함된 ID/PW로 인증 수행
- Windows 2016 이상 버전에서는 보안상 이슈로 인해 Telnet 서버 설치 제공하지 않음 [W-34]

| 취약 | 세션 유휴 타임아웃이 설정되지 않은 경우 | 방치된 활성 세션을 통해 세션 하이재킹 공격이 가능하다 |

### 참고 문서
- Administrator는 해당 정책을 활성화하여 컴퓨터가 비활성 SMB Session을 중단하는 시점을 제어할 수 있으며, 클라이언트를 다시 시작하면 해당 Session은 자동으로 다시 연결됨. 해당 정책의 값을 0으로 설정하면 가능한 한 빨리 유휴 Session 연결은 끊어지며, 최댓값은 99999(208일)로 사실상 정책 설정 해제를 의미함
- SMB(서버 메시지 블록): LAN이나 컴퓨터 간의 통신에서 데이터 송수신을 하기 위한 프로토콜 [W-56]

## [W-77] LAN Manager 인증 수준 (계정_스크립트)

### 개요
| 항목 | 내용 |
|------|------|
| 점검 코드 | W-77 |
| 항목명 | LAN Manager 인증 수준 |
| 위험도 | 중 (Medium) |
| 범주 | 네트워크 인증 |

### 점검 목적
LAN Manager(LM) 인증은 구형 Windows에서 사용되던 약한 해시 알고리즘입니다. LM 인증을 허용한 경우 패스워드 해시가 약하게 생성되어 오프라인 크래킹 공격에 취약해집니다. NTLMv2 이상만 허용하여 보안성을 강화해야 합니다.

### 점검 기준
| 구분 | 기준 | 설명 |
|------|------|------|
| 안전 | LAN Manager 인증이 비활성화되고 NTLMv2 이상만 사용되는 경우 | 약한 LM 해시 생성이 차단되어 패스워드 크래킹 난이도가 크게 증가한다 |
| 취약 | LAN Manager 인증이 허용되는 경우 | LM 해시를 통해 패스워드가 쉽게 크래킹될 수 있다 |

### 참고 문서
- LAN Manager는 네트워크를 통한 파일 및 프린터 공유 등과 같은 작업 시 인증을 담당. NTLMv2는 Windows 2000, 2003, XP 이상에서 지원되며, Windows 98, NT 버전과 통신하면 패치를 설치해야 함 [W-59]

#### `W-82` `중` Windows 인증 모드 사용

## [W-82] Windows 인증 모드 사용 (계정_스크립트)

### 개요
| 항목 | 내용 |
|------|------|
| 점검 코드 | W-82 |
| 항목명 | Windows 인증 모드 사용 |
| 위험도 | 중 (Medium) |
| 범주 | 인증 설정 |

### 점검 목적
SQL Server 등 데이터베이스에서 혼합 인증 모드(Windows 인증 + SQL 인증)가 활성화된 경우, SQL 계정의 패스워드가 약하게 설정되거나 기본값으로 운영될 위험이 있습니다. Windows 인증 모드만 사용하여 도메인 계정의 중앙 집중식 인증 및 패스워드 정책을 적용해야 합니다.

### 점검 기준
| 구분 | 기준 | 설명 |
|------|------|------|
| 안전 | Windows 인증 모드만 사용하도록 설정된 경우 | 도메인 계정의 중앙 집중식 인증 및 패스워드 정책이 적용되어 계정 보안이 강화된다 |
| 취약 | 혼합 인증 모드(Windows + SQL 인증)가 활성화된 경우 | 별도 관리되는 SQL 계정의 약한 패스워드로 인한 DB 무단 접근 위험이 있다 |

### 참고 문서
- 데이터베이스 엔진 인증 모드에는 Windows 인증 모드와 SQL Sever가 있는 혼합 모드 두 가지 구성이 있음. Windows 인증 모드 선택 시 SQL Sever 인증을 위해서 설치 프로그램은 sa라는 비활성화된 계정을 생성하고, 이 계정은 혼합 모드를 사용함으로써 활성화됨. sa 계정은 일반 사용자들에게 잘 알려진 만큼 쉽게 공격의 대상이 될 수 있으므로 꼭 필요하지 않은 경우 비활성화하고, 활성화해야 할 경우 강력한 암호 체계를 사용해야 함
- Windows 인증은 kerberos 보안 프로토콜을 사용하며, 강력한 암호 정책을 적용하여 적합한 복잡성 수준을 유지함. 또한, 계정 잠금 및 암호만료를 지원하고 SQL 서버가 Windows에서 제공하는 자격 증명을 신뢰한 트러스트 연결을 사용하기 때문에 Windows 인증 모드 사용을 권고함
- sa 계정: 데이터베이스 서버 설치 시 자동으로 생성되며 DB 서버 관리자 계정
- kerberos 보안 프로토콜: 개방된 컴퓨터 네트워크 내에서 서비스 요구를 인증하기 위한 보안 시스템 [D-16]
