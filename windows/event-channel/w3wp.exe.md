보내주신 윈도우 디펜더 로그 정리 방식에 맞춰, **IIS(Internet Information Services)의 핵심 프로세스인 `w3wp.exe`(Worker Process)**와 관련된 주요 이벤트 로그 및 모니터링 포인트를 정리해 드립니다.

`w3wp.exe` 관련 로그는 주로 **[이벤트 뷰어] -> [Windows 로그] -> [System]** 또는 **[Application]**에서 확인할 수 있으며, 소스(Source)가 **WAS(Windows Process Activation Service)** 또는 **W3SVC(World Wide Web Publishing Service)**인 항목을 중점적으로 보아야 합니다.

---

### 1. 프로세스 수명 주기 및 상태 (Lifecycle & Status)

응용 프로그램 풀(App Pool)이 시작되거나 중단되는 등 프로세스의 생존 주기와 관련된 이벤트입니다.

| 이벤트 ID | 소스 | 설명 및 의미 |
| --- | --- | --- |
| **5059** | WAS | **애플리케이션 풀 시작**: 특정 응용 프로그램 풀이 성공적으로 시작되었습니다. |
| **5074** | WAS | **워커 프로세스 종료**: 요청된 작업 완료 후 또는 유휴 시간 초과로 인해 프로세스가 종료되었습니다. |
| **5076** | WAS | **프로세스 재생(Recycle) 시작**: 설정된 일정, 메모리 제한, 또는 수동 요청에 의해 재생이 시작되었습니다. |
| **5077** | WAS | **프로세스 재생 완료**: 기존 프로세스가 종료되고 새로운 `w3wp.exe`가 성공적으로 생성되었습니다. |
| **5186** | WAS | **활성 작업자 프로세스 없음**: 응용 프로그램 풀에 요청을 처리할 작업자 프로세스가 없는 상태를 나타냅니다. |

### 2. 비정상 종료 및 오류 (Errors & Crashes)

웹 서비스 중단의 원인을 파악할 때 가장 먼저 확인해야 하는 중요한 섹션입니다.

| 이벤트 ID | 소스 | 설명 및 의미 |
| --- | --- | --- |
| **5011** | WAS | **프로세스 응답 없음**: `w3wp.exe`가 핑(Ping) 응답에 실패하여 WAS에 의해 강제 종료되었습니다. (데드락 가능성) |
| **5013** | WAS | **시간 초과**: 워커 프로세스가 지정된 시간 내에 요청을 처리하지 못해 강제로 중지되었습니다. |
| **5117** | WAS | **급격한 실패 보호(Rapid Fail Protection)**: 짧은 시간 내에 프로세스가 반복적으로 충돌하여 응용 프로그램 풀이 자동으로 중단(Stopped)되었습니다. |
| **1000** | Application Error | **Application Crash**: `w3wp.exe` 자체가 예외 오류로 인해 비정상 종료되었습니다. (모듈 이름 및 오류 주소 포함) |
| **1309** | ASP.NET | **미처리 예외**: ASP.NET 애플리케이션 내에서 처리되지 않은 런타임 오류가 발생하여 로그에 기록되었습니다. |

### 3. 성능 및 리소스 경고 (Performance & Resources)

서버 부하나 설정 임계치 초과와 관련된 로그입니다.

| 이벤트 ID | 소스 | 설명 및 의미 |
| --- | --- | --- |
| **5010** | WAS | **상태 확인 실패**: 프로세스가 활성 상태이지만 통신에 문제가 있음을 감지했습니다. |
| **5111** | WAS | **메모리 제한 초과**: 설정된 가상/전용 메모리 사용량을 초과하여 프로세스 재생이 트리거되었습니다. |
| **5091** | WAS | **큐 가득 참**: 응용 프로그램 풀의 대기 큐(Queue)가 가득 차서 `503 Service Unavailable` 오류가 발생하기 시작합니다. |

### 💡 문제 해결을 위한 추가 팁

**1. HTTP.sys 로그 확인**
`w3wp.exe`까지 도달하지 못하고 네트워크 단에서 발생하는 오류(`503`, `400` 등)는 아래 경로의 텍스트 로그를 확인해야 합니다.

* 경로: `%SystemRoot%\System32\LogFiles\HTTPERR`

**2. PowerShell로 특정 App Pool 로그 검색**
특정 응용 프로그램 풀(예: 'MyAppPool')에 발생한 에러만 빠르게 필터링할 때 유용합니다.

```powershell
Get-WinEvent -LogName System | Where-Object { $_.Message -like "*MyAppPool*" } | Select-Object TimeCreated, Id, Message | Out-GridView

```

**3. 디버그 진단 (Debug Diagnostics)**
이벤트 ID 1000번과 같이 원인 불명의 Crash가 반복된다면, Microsoft의 **DebugDiag** 도구를 사용하여 메모리 덤프를 분석하는 것이 가장 정확합니다.

---

네, 충분히 가능합니다. `w3wp.exe`는 웹 서버의 권한을 직접 가지고 실행되는 프로세스이기 때문에, 해커가 웹 취약점(SQL Injection, File Upload 등)을 통해 이 프로세스의 제어권을 획득하면 서버 내부망 침투나 데이터 유출의 핵심 도구로 악용될 수 있습니다.

해커가 `w3wp.exe`를 악용하는 주요 패턴을 범주별로 정리해 드립니다.

---

### 1. 웹셸(WebShell) 설치 및 원격 제어

해커가 가장 선호하는 방식입니다. 웹 서버에 악성 스크립트를 업로드한 후, `w3wp.exe`를 통해 시스템 명령어를 실행합니다.

* **임의 명령 실행:** `w3wp.exe` 프로세스의 자식 프로세스로 `cmd.exe`나 `powershell.exe`를 실행하여 서버를 제어합니다.
* **권한 상승 시도:** 기본적으로 `IIS AppPool\<AppPoolName>`이라는 저권한 계정으로 구동되지만, 시스템 취약점을 이용하여 `LocalSystem`이나 `Administrator` 권한 탈취를 시도합니다.

### 2. C2 서버와의 통신 및 데이터 유출 (Exfiltration)

질문하신 내용처럼 `w3wp.exe`가 외부로 접속하게 만드는 행위입니다.

* **리버스 커넥션 (Reverse Shell):** 보안 장비가 외부에서 들어오는 접속은 차단하지만, 내부에서 나가는 접속(Outbound)은 허용하는 점을 악용합니다. `w3wp.exe`가 해커의 C2(Command & Control) 서버로 직접 연결을 시도하게 합니다.
* **데이터 외부 전송:** 서버 내부에 저장된 고객 정보, 데이터베이스 덤프 파일, 소스 코드 등을 HTTP POST 요청이나 FTP 등을 통해 외부로 빼돌립니다.
* **추가 악성코드 다운로드:** 다른 공격 도구를 서버로 가져오기 위해 외부 URL에서 파일을 다운로드하는 통로로 사용합니다.

### 3. 내부망 스캐닝 및 측면 이동 (Lateral Movement)

침투한 웹 서버를 거점(Pivot)으로 삼아 사내 네트워크의 다른 서버를 공격합니다.

* **내부 스캔:** `w3wp.exe` 권한으로 내부망의 다른 IP와 포트가 열려 있는지 확인합니다.
* **공격 전파:** 내부 DB 서버나 파일 서버의 취약점을 공격하여 감염 범위를 넓힙니다.

### 4. 파일리스(Fileless) 공격 및 지속성 유지

디스크에 흔적을 남기지 않고 메모리상에서만 동작하여 탐지를 회피합니다.

* **메모리 인젝션:** 정상적인 `w3wp.exe` 프로세스 메모리 영역에 악성 코드를 삽입하여 실행합니다. 이는 일반적인 백신 탐지를 우회하는 경우가 많습니다.
* **프로세스 중지 방해:** 디펜더의 변조 방지 기능이 꺼져 있다면, 자신의 활동을 숨기기 위해 보안 로그 작성을 방해하거나 프로세스를 조작합니다.

---

### 🛡️ 탐지를 위한 핵심 모니터링 포인트

해킹 여부를 판단하기 위해 이벤트 로그와 네트워크 상태에서 다음 징후를 살펴봐야 합니다.

| 모니터링 항목 | 이상 징후 (Red Flag) |
| --- | --- |
| **자식 프로세스** | `w3wp.exe`가 `cmd.exe`, `powershell.exe`, `whoami.exe`, `net.exe` 등을 실행하는 경우 |
| **네트워크 연결** | 웹 서비스 포트(80, 443) 외에 생소한 외부 IP로 대량의 Outbound 트래픽이 발생하는 경우 |
| **파일 생성** | `inetpub\wwwroot` 경로에 알 수 없는 `.aspx`, `.php`, `.jsp` 파일이 생성되는 경우 |
| **이벤트 로그** | **Event ID 5011, 5117** 등이 단시간에 집중 발생(공격 시도로 인한 프로세스 충돌 가능성) |

**조치 팁:** 가장 효과적인 방어 중 하나는 **"최소 권한 원칙"**입니다. 응용 프로그램 풀 계정이 필요 이상의 폴더에 접근하지 못하도록 NTFS 권한을 제한하고, 방화벽에서 웹 서버가 외부로 나가는 Outbound 포트를 화이트리스트 기반으로 차단하는 것이 좋습니다.

혹시 현재 서버에서 `w3wp.exe`가 특정 외부 IP로 접속하는 이력을 확인하고 싶으신가요? **PowerShell을 이용해 실시간 네트워크 연결을 확인하는 방법**을 안내해 드릴 수 있습니다.
